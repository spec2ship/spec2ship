# Output Formats by Workflow Type

Guide for generating output documents after roundtable completion.

## Specs Workflow → Requirements Document

**Output file**: `docs/specifications/requirements.md`

**Format**: SRS-lite (simplified Software Requirements Specification)

**Sections**:
1. **Introduction**: Project overview from CONTEXT.md
2. **Functional Requirements**: From roundtable consensus on core-functional topics
3. **Non-Functional Requirements**: From roundtable consensus on NFR topics
4. **Acceptance Criteria**: From roundtable discussion on acceptance-criteria
5. **Out of Scope**: Explicitly excluded items from roundtable

**Reference skill**: `iso25010-requirements` for quality attributes

**Template**:
```markdown
# Requirements: {topic}

## 1. Introduction
{from CONTEXT.md objectives and scope}

## 2. Functional Requirements
{from consensus items tagged "functional"}

## 3. Non-Functional Requirements
{from consensus items tagged "nfr"}

## 4. Acceptance Criteria
{from roundtable discussion}

## 5. Out of Scope
{explicitly excluded items}

---
*Generated from roundtable session: {session-id}*
```

## Design Workflow → Architecture Documentation

**Output files**:
- `docs/architecture/README.md` - Overview
- `docs/architecture/components.md` - Building blocks
- `docs/decisions/{timestamp}-{slug}.md` - ADRs for key decisions

**Format**: arc42-lite (simplified arc42)

**Sections for README**:
1. **Introduction & Goals**: From CONTEXT.md
2. **Constraints**: From roundtable consensus
3. **Context & Scope**: System boundaries
4. **Solution Strategy**: Key architectural decisions
5. **Building Block View**: Component overview (link to components.md)

**Reference skills**: `arc42-templates`, `madr-decisions`

**ADR Template** (for each key decision):
```markdown
# {YYYYMMDD-HHMMSS}-{decision-slug}

## Status
Accepted

## Context
{from roundtable discussion - what problem needed solving}

## Decision
{from roundtable consensus - what was decided}

## Consequences
{from roundtable discussion - trade-offs and implications}

---
*Generated from roundtable session: {session-id}*
```

## Brainstorm Workflow → Summary Document

**Output file**: `.s2s/sessions/{session-id}-summary.md`

**Format**: Disney Strategy Summary

**Sections**:
1. **Executive Summary**: 2-3 sentence overview
2. **Dreamer Phase Ideas**: All ideas generated (consensus items)
3. **Realist Assessment**: Feasibility notes, effort estimates
4. **Critic Risks**: Identified risks and mitigations
5. **Recommended Next Steps**: Actionable items

**No reference skill required** - free-form creative output

**Template**:
```markdown
# Brainstorm: {topic}

**Session**: {session-id}
**Date**: {date}
**Strategy**: {strategy}
**Participants**: {count} ({list})
**Rounds**: {total_rounds}

---

## Executive Summary
{2-3 sentence synthesis of key outcomes}

## Dreamer Phase Ideas
{list of all ideas from dreamer rounds}

## Realist Assessment
{feasibility notes from realist rounds}

## Critic Risks
{risks identified and mitigations proposed}

## Recommended Next Steps
{actionable items}

---
*Generated by Spec2Ship roundtable*
```

## Output Type Selection

The facilitator suggests `output_type` in final synthesis based on:

| Workflow | Default output_type | Alternative |
|----------|---------------------|-------------|
| specs | `requirements` | `summary` |
| design | `architecture` | `adr` |
| brainstorm | `summary` | - |

User can override via `--output-type` flag.
